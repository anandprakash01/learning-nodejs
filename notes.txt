

/* ===========================================================================
THE ARCHITECT'S NOTES: package.json (Ecosystem & Execution Physics)
===========================================================================
This is the root contract for the V8 engine. It tells Node.js exactly how 
to allocate memory and which C++ bindings/libraries to load into the heap.

The "Mistake": 
Amateurs use `node server.js` directly in production. This leaves the V8 engine 
without strict memory constraints, leading to silent Out-Of-Memory (OOM) 
crashes when the Garbage Collector fails to keep up with incoming network streams.
They also fail to lock engine versions, causing libuv behavior to change 
unpredictably across different developer machines.

The "Enterprise Way": 
1. V8 Heap Boundaries: We use `--max-old-space-size=2048` in the start script 
   to strictly cap the V8 heap at 2GB. If it hits this limit, it crashes 
   predictably rather than locking up the entire OS host.
2. Native Debugging: We include a native `--inspect` debug script so your IDE 
   (VS Code) can attach its debugger directly to the libuv event loop for 
   step-through memory analysis.
3. Engine Locking: We enforce Node.js >= 20.0.0 to guarantee specific 
   Stream API and Worker Thread capabilities.
===========================================================================
*/

1. npm i nodemon (The Production Binding)

What it does: It downloads the nodemon binaries into your node_modules folder and saves the exact version number under the "dependencies" object in your package.json.

The Physics: By default, npm assumes everything you install is strictly required to run your business logic in production.

The Enterprise Mistake: If you do this with nodemon, when your DevOps pipeline builds your production Docker container, it will install nodemon. nodemon contains thousands of lines of code dedicated to watching OS file systems. In a production environment, your files do not change. Shipping this to production bloats your server's memory footprint and increases the surface area for security vulnerabilities.

2. npm install --save-dev nodemon (The Development Isolation)

What it does: npm install and npm i are the exact same command. However, adding the --save-dev (or -D) flag instructs the Node package manager to isolate this tool. It saves it under the "devDependencies" object in package.json.

The Physics: This creates a strict environmental boundary. It tells the ecosystem: "This tool is only for my local machine."

The Enterprise Way: When an enterprise CI/CD pipeline deploys your code to AWS or Kubernetes, it runs npm install --omit=dev. This command explicitly tells Node.js to ignore everything in the "devDependencies" list. This guarantees that your production V8 engine remains perfectly lean, containing only the absolute minimum code required to process TCP network streams.




1. The Standard MVC Architecture (The First Tree)
The first tree is the industry-standard Model-View-Controller (MVC) pattern adapted for an API.

The Flow: Router -> Controller -> Service -> Mongoose Model.

The Physics: Everything happens synchronously within the main V8 thread. If a user registers, the Service hashes the password, saves it to MongoDB, and sends the welcome email before responding to the user.

The Limit: This is perfectly fine for thousands of users. But if you hit 100,000 users, sending emails during the HTTP request will block the libuv thread pool, and your server will crash.

2. The Distributed Enterprise Architecture (The Second Tree)
The second tree is a Distributed, Event-Driven Architecture. This is what tech giants use. It introduces extreme decoupling to protect the V8 engine at all costs.

Here is what the extra folders actually do:

📁 repositories/ (The Database Boundary): In standard MVC, your services talk directly to Mongoose. In this advanced tree, the Repository pattern abstracts Mongoose entirely. If you ever want to switch from MongoDB to PostgreSQL, you only rewrite the Repository. The Service logic never changes.

📁 queues/ & 📁 jobs/ (Background Workers): Instead of sending an email while the user waits, the Controller pushes an event into a Redis memory queue (like BullMQ) and immediately responds 201 Created to the user. A completely separate background CPU thread picks up the job and sends the email silently.

📁 events/ (Internal Pub/Sub): This brings back the EventEmitter physics we discussed earlier, allowing different parts of your system to communicate without importing each other's files.
