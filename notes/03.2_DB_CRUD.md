# Mongoose: CRUD Operations

Mongoose provides a massive arsenal of methods to perform CRUD (Create, Read, Update, Delete) operations. These are primarily executed directly on your uppercase Model (e.g., `User.find()`), though a few are executed on the individual document (e.g., `user.save()`).

---

## Create (Inserting Data)

These methods take JavaScript objects and turn them into new documents in your MongoDB collection.

### 1. `new Model()` + `.save()`

- **How it works:** You create an instance of the model in memory, modify it if needed, and then call `.save()` to write it to the database.
- This is the most common way to create a single document. It fully triggers all validation and `pre('save')` middleware.

```javascript
const newUser = new User({username: "Alice", age: 28});
await newUser.save();
```

### 2. `Model.create(docs)`

- **How it works:** A shortcut for creating and saving one or multiple documents at once.
- Cleaner syntax when you don't need to manipulate the document in memory before saving.

```javascript
// Creates one
await User.create({username: "Bob"});

// Creates multiple (pass an array)
await User.create([{username: "Charlie"}, {username: "Dave"}]);
```

### 3. `Model.insertMany(docs)`

- **How it works:** Inserts a massive array of documents directly into MongoDB.
- **Performance:** It is much faster than `Model.create()` for large datasets, but it bypasses `pre('save')` hooks (though it still validates).

If you use `Model.insertMany([doc1, doc2, doc3])` to create a bunch of records at once, it does not trigger `.pre("save")` for each document.

---

## Read (Fetching Data)

These methods query the database and return documents. They return a Mongoose `Query` object, which you typically `await` to get the actual results.

### 1. `Model.find(filter)`

- **Returns:** An Array of documents (or an empty array `[]` if none match).

```javascript
const adults = await User.find({age: {$gte: 18}});
```

### 2. `Model.findOne(filter)`

- **Returns:** A single Document (the first one it finds), or `null` if nothing matches.

### 3. `Model.findById(id)`

- **Returns:** A single Document or `null`.
- **Why use it:** A convenience wrapper around `.findOne({ _id: id })`.

```javascript
const user = await User.findById("60d5ec49f1b2c42d...");
```

## Update (Modifying Data)

If you use `.updateOne()`, `.updateMany()`, or `.findOneAndUpdate()`, your `.pre("save")` hook will be completely ignored.

**Why `.update()` bypasses `.pre("save")`**
Mongoose has two entirely different types of middleware: Document Middleware and Query Middleware.

- **`.save()` uses Document Middleware:** When you use `.save()`, Mongoose pulls the actual document out of the database and loads it into your Node.js memory. Because it is in memory, Mongoose can easily run your `.pre("save")` function on it before sending it back to the database.
- **`.update()` uses Query Middleware:** When you use an update command, Mongoose does not pull the document into memory. Instead, it sends a raw command directly to the MongoDB server saying, "Hey database, find this ID and change this field." Because the document never touches your Node.js server, the `.pre("save")` hook is completely blind to it.

**⚠️ The Golden Rule of Updating:** There are two ways to update data.

1. Fetch the document, change it, and call `.save()`. (Best for triggering hooks).
2. Tell the database to update it directly. (Faster, but skips `save` hooks).

### 1. `Model.updateOne(filter, update)` / `Model.updateMany()`

- **How it works:** It tells the database to find matching documents and apply changes.
- **Returns:** An acknowledgment object (e.g., `{ modifiedCount: 1 }`), not the document itself.

```javascript
await User.updateMany({role: "user"}, {$set: {active: true}});
```

### 2. `Model.findByIdAndUpdate(id, update, options)` / `Model.findOneAndUpdate()`

- **How it works:** This finds a specific document, updates it, and returns the document.
- **The Gotcha:** By default, it returns the document as it was before the update. To get the newly updated document, you must pass `{ new: true }`.

```javascript
const updatedUser = await User.findByIdAndUpdate(
  "_id",
  {age: 29},
  {new: true, runValidators: true},
);
// Always pass runValidators!
```

## Delete (Removing Data)

Just like saving, developers often write a `.pre("remove")` or `.pre("deleteOne")` hook on their document. For example, if you delete a User, you might want a hook to automatically delete all the Events that user created.

These methods will ignore your document deletion hooks:

- `Model.deleteOne()`
- `Model.deleteMany()`
- `Model.findByIdAndDelete()`
- `Model.findOneAndDelete()`

**How to fix the delete trap:** If you need to trigger cascading deletes (like deleting a user's events when the user is deleted), you have to fetch the user first:

```javascript
const user = await User.findById(userId);
await user.deleteOne();
```

### 1. `Model.deleteOne(filter)` / `Model.deleteMany(filter)`

- **How it works:** It deletes documents directly in the database.
- **Returns:** An acknowledgment object (e.g., `{ deletedCount: 5 }`).

```javascript
await User.deleteMany({active: false});
```

### 2. `Model.findByIdAndDelete(id)` / `Model.findOneAndDelete(filter)`

- **How it works:** It finds a document, deletes it, and returns the deleted document (useful if you need to know exactly what was destroyed).

```javascript
const deletedUser = await User.findByIdAndDelete(_id);
```

Under the hood, `findByIdAndDelete` and `findByIdAndUpdate` are simply convenience wrappers (syntactic sugar) for `findOneAndDelete` and `findOneAndUpdate`.
Because they are literally the exact same function under the hood, they take the exact same options objects.

`findByIdAndRemove()`
MongoDB officially deprecated remove in favor of delete. So today, you should always stick to `findByIdAndDelete` and `findOneAndDelete`!

## Other / Advanced Methods (The Power Tools)

When basic CRUD isn't enough, Mongoose provides these tools for analytics, counting, and complex operations.

### 1. `Model.countDocuments(filter)`

- **Returns:** It returns the exact number of documents that match a query.
- **Why use it:** Pagination (knowing how many total pages you have) or basic statistics.

```javascript
const totalAdmins = await User.countDocuments({role: "admin"});
```

### 2. `Model.exists(filter)`

- **Returns:** It returns an object with the `_id` if a matching document exists, or `null`.
- **Why use it:** It is much faster than `.findOne()` if you only need to know if a record is there, but don't actually need its data.

### 3. `Model.distinct(field, filter)`

- **Returns:** It returns an array of all unique values for a specific field.
- **Why use it:** Great for populating dropdown menus (e.g., "Get all unique categories currently used by products").

```javascript
const activeCities = await User.distinct("city", {active: true});

// Returns: ['New York', 'London', 'Tokyo']
```

### 4. `Model.aggregate(pipeline)`

`aggregate(pipeline)`: Bypasses normal Mongoose query logic and runs raw MongoDB aggregation. It returns an Aggregate object, not a Query.

- This is MongoDB's heavy lifter. You pass it an array of stages (like `$match`, `$group`, `$lookup`, `$sort`) to process data step-by-step.
- **Why use it:** Complex reporting, calculating averages/sums, or joining collections together.
- **What it returns:** It does not return raw documents. It returns a brand new, custom JSON object that you just mathematically invented on the fly.

```javascript
// Find the average age of users grouped by their role
const stats = await User.aggregate([
  {
    $group: {
      _id: "$role",
      averageAge: {$avg: "$age"},
    },
  },
]);
```
