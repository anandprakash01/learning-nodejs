# Shield 2: Express Rate Limit (The Door Bouncer)

We build the counting mechanism. We define a "window" of time, and a "max" number of requests allowed in that window.

You shouldn't apply the exact same rate limit to every single API endpoint.
However, the best practice is to have a two-tiered approach: a loose global limiter for safety, and strict, specific limiters for sensitive endpoints.

Here is a breakdown of how you should think about applying rate limiters across your architecture:

### 1. The Global Safety Net (Apply Everywhere)

You should have a basic, high-capacity rate limiter applied globally across your entire application.

- **Purpose:** To prevent massive DDoS attacks, stop runaway scripts (like a user accidentally writing an infinite loop that calls your API), and keep your servers from crashing.
- **Example Limit:** 1,000 requests per minute per IP address.

### 2. Route-Specific Limiters (Apply Strategically)

Different actions have different costs and security risks. You need to apply custom, much stricter limiters to specific endpoints based on what they do.

- **Authentication & Security (Extremely Strict):**
  - **Endpoints:** `/login`, `/forgot-password`, `/verify-otp`
  - **Why:** To prevent brute-force password guessing and credential stuffing.
  - **Limit:** 5 requests per 15 minutes per IP or Email.
- **Heavy Computations / Data Exports (Strict):**
  - **Endpoints:** `/export-csv`, `/generate-report`, AI generation endpoints.
  - **Why:** These consume massive CPU and memory. A user spamming this can slow down the database for everyone else.
  - **Limit:** 2 requests per minute per User ID.
- **Monetization / Tiered Limits (Business Logic):**
  - **Endpoints:** Core SaaS features (e.g., `/api/v1/search`)
  - **Why:** You want to enforce billing plans.
  - **Limit:** "Free Tier" gets 100 req/day; "Pro Tier" gets 10,000 req/day (Limited by API Key, not IP).

## The Brain of the Rate Limiter

How does it count?
By default, `express-rate-limit` is completely blind to your database. It does not know what a "password" is, and it does not know if a login was successful or if it failed.

**Limit Requests from the same IP:**

```javascript
const limiter = rateLimit({
  max: 100, // Limit each IP to 100 requests...
  windowMs: 60 * 60 * 1000, // ...per 1 hour (in milliseconds)
  message: "Too many requests from this IP, please try again in an hour!", // The 429 message
});

// We apply this limiter ONLY to routes that start with /api
app.use("/api", limiter);
```

### How the Rate Limiter Physically Works (The Fixed Window)

Under the hood, `express-rate-limit` uses an algorithm called a `Fixed Window Counter`.

When you boot up your server, the Rate Limiter allocates a tiny, empty dictionary object in your Node.js RAM: `const hits = {};`

- **The Arrival:** A request hits your server. The Limiter looks at the TCP socket and extracts the IP address (e.g., `192.168.1.5`).
- **The Dictionary Lookup:** It checks the `hits` dictionary.
  - If the IP is not there, it adds/logs it: `hits['192.168.1.5'] = 1`.
  - If it is there, it increments the number: `hits['192.168.1.5'] += 1`.
- **The Bouncer Check:** It compares that number to your `max` variable. If it exceeds the max, it throws an error and drops the connection.
- **The Timer (The Window):** When you set `windowMs: 15 * 60 * 1000` (15 minutes), the Limiter starts a V8 stopwatch. Exactly 15 minutes after that IP made its first request, the Limiter completely deletes that IP from the dictionary. Their slate is wiped clean.

Because it sits before the controller, it counts EVERY request. Good passwords, bad passwords, successful data fetches, 404 errors—it counts them all.

**Why do we design it this way?**
Because this specific shield is a Volumetric Shield. Its only job is to protect your V8 CPU and RAM from being crushed by sheer volume (a DDoS attack). If a legitimate user refreshes the page 100 times in one minute, we want to block them, because they are consuming too much CPU, even if they are logged in!

## The Enterprise Fix (Account Lockout)

How do we punish hackers trying bad passwords, without punishing normal users browsing the site?

In Enterprise Architecture, we split the defense into two completely separate layers:

**Layer 1: The Global IP Limiter (The Bouncer)**
This is the tool we above installed. It limits all traffic from an IP address to a high number (e.g., 1,000 requests per hour). It stops brute-force DDoS attacks from melting the server.

**Layer 2: The Specific Login Limiter (The Detective)**
We create a second, much stricter Rate Limiter, and we attach it only to the Login route.

```javascript
import rateLimit from "express-rate-limit";

// => This ONLY counts requests `from the same IP` hitting the `/login` endpoint
export const loginLimiter = rateLimit({
  max: 5, // Only 5 login attempts allowed...
  windowMs: 15 * 60 * 1000, // ...per 15 minutes!
  message: "Too many login attempts from this IP, please try again in 15 minutes.",
});

app.use("/api/users/login", limiter, handleLogin);
```

This library has a massive arsenal of advanced configurations that solve the exact problems.

### The Smart Counter (`skipSuccessfulRequests`)

It count the correct passwords too. But you can make the Limiter incredibly smart.

```javascript
export const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // ...per 15 min
  max: 5,
  skipSuccessfulRequests: true,
});
```

If you set this to `true`, the Limiter watches the final HTTP status code your controller sends. If your controller sends `200 OK` (success), the Limiter reaches into its RAM and deletes that hit from the counter. It will only count requests that result in 400, 401, or 500 errors! Normal users are never punished.

### The Standard Headers (`standardHeaders` & `legacyHeaders`)

When a Rate Limiter blocks a user, the frontend usually wants to show a countdown timer: "Try again in 14 minutes." But how does the frontend know how much time is left?

```javascript
export const apiLimiter = rateLimit({
  windowMs: 60 * 60 * 1000, // ... per 60 min
  max: 100,
  standardHeaders: true, // Return rate limit info in the `RateLimit-*` headers
  legacyHeaders: false, // Disable the deprecated `X-RateLimit-*` headers
});
```

This forces the Limiter to attach mathematical metadata to the HTTP response. It sends headers like `RateLimit-Remaining: 99` and `RateLimit-Reset: 1689234000`. The React frontend can read these and build a live UI timer.

### The Botnet Defense (`keyGenerator`)

By default, the Limiter counts the `req.ip`. But what if a hacker has 10,000 different computers (a botnet), and they try 10,000 different passwords on your email address? Because every IP is unique, the `max: 5` limit will never trigger!

We can rewrite the Limiter's brain to count the Email Address instead of the IP.

```javascript
export const botnetLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // ...per 15 min
  max: 5,
  // 🚨 OVERWRITE THE DICTIONARY KEY
  keyGenerator: req => {
    // Instead of using req.ip, we use the email they typed in!
    return req.body.email;
  },
});
```

Now, the RAM dictionary looks like this: `hits['admin@startup.com'] = 5`. It does not matter if the hacker changes their IP address a million times. The Limiter is locked onto the email target and will shut the door.

`keyGenerator` is a native, built-in property of the `express-rate-limit` configuration object.
When a request hits the Rate Limiter, the limiter needs a "Name Tag" to put in its RAM dictionary so it knows who to count.

- **The Default:** If you don't write `keyGenerator`, the limiter uses its own internal default function, which is literally just: `(req) => { return req.ip }`.
- **The Override:** By writing `keyGenerator: (req) => req.body.email`, you intercept the limiter's internal brain, and use whatever string I return as the Name Tag.

### The Hybrid `keyGenerator` (IP + Email)

The Rate Limiter dictionary stores data as Key: Value.
If you want to track a combination of two things, you simply stitch them together into a single string using template literals.

```javascript
export const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 5,
  skipSuccessfulRequests: true,

  // 🚨 THE HYBRID KEY
  keyGenerator: req => {
    // If the IP is 192.168.1.5 and the email is admin@test.com,
    // the V8 dictionary creates this key: "192.168.1.5_admin@test.com"
    return `${req.ip}_${req.body.email}`;
  },
  message: "Too many login attempts from this IP for this account.",
});
```

**The Architectural Consequence:**
You must understand what this specific mathematical choice does to your security:

- If you only use email (`req.body.email`), the hacker gets exactly 5 tries total across the entire planet. (Maximum security, but could lock out the real user if the hacker keeps trying).
- If you use the hybrid (`req.ip` + `req.body.email`), the hacker gets 5 tries per IP address. If they have a botnet of 1,000 computers, they get 5,000 total attempts at the password.

Because the hybrid method is slightly weaker against botnets, we must build Shield 3 (The Vault) to guarantee the account physically locks down at the database level.

# Shield 3: Database Account Lockout (The Vault)

Even with an IP Limiter, a smart hacker will use a Botnet (10,000 different computers with 10,000 different IPs) to try 1 password on your account. The IP Limiters will never catch them because each IP only makes 1 request!

This is the ultimate Enterprise defense. If a hacker somehow bypasses your RAM-based Rate Limiters, the MongoDB hard drive will physically refuse to let anyone log into the account.

### Step 1: Upgrading the State (`User.js`)

To stop this, we add physics to the User Schema itself:

```javascript
failedLoginAttempts: {
  type: Number,
  default: 0,
  select: false // Hide this from the frontend response
},
accountLockedUntil: {
  type: Date,
  select: false // Hide this from the frontend response
}
```

### Step 2: The Vault Logic (`userController.js`)

```javascript
// 1. Fetch the user AND explicitly pull the hidden security fields into RAM
const user = await User.findOne({email}).select(
  "+password +failedLoginAttempts +accountLockedUntil",
);

if (!user) {
  throw new AppError("Incorrect email or password", 401);
}

// 🚨 2. THE VAULT CHECK: Is the door already locked?
if (user.accountLockedUntil && user.accountLockedUntil > Date.now()) {
  const timeLeft = Math.ceil((user.accountLockedUntil - Date.now()) / 60000);
  throw new AppError(
    `Account temporarily locked due to multiple failed attempts. Try again in ${timeLeft} minutes.`,
    403,
  );
}

// 3. THE PASSWORD CHECK
const isMatch = await user.matchPassword(password);

if (!isMatch) {
  // 🚨 4. THE PENALTY: Wrong password. Increment the strikes.
  user.failedLoginAttempts += 1;

  if (user.failedLoginAttempts >= 5) {
    // Strike 5: Set the lock timer for 15 minutes in the future
    user.accountLockedUntil = Date.now() + 15 * 60 * 1000;
  }

  // Save the penalty to the database without triggering validation rules
  await user.save({validateBeforeSave: false});

  throw new AppError("Incorrect email or password", 401);
}

// 🚨 5. THE SUCCESS RESET: The real user logged in successfully!
// Wipe the slate clean so they don't get locked out next time.
user.failedLoginAttempts = 0;
user.accountLockedUntil = undefined;
await user.save({validateBeforeSave: false});

// ... Generate token and send response ...
```

## The Conclusion

- Use `helmet` to protect the React Browser.
- Use a **Global Rate Limiter** (1,000 req/hr) to protect the Node.js CPU from volumetric attacks.
- Use a **Strict Route Limiter** (5 req/min) specifically on the login route.
- Use **Database State** (`failedLoginAttempts`) to permanently lock an account under attack, regardless of what IP address the hacker uses.
