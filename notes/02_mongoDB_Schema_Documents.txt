
➖➖➖ MongoDB is a NoSQL database. At its core, it is completely lawless.
MongoDB is a database written in C++. It stores data in a binary format called BSON. Node.js server speaks JavaScript. They do not natively understand each other.

➖➖➖ Mongoose is an ODM (Object Data Modeling) library. It acts as the translator. It takes the JavaScript objects, formats them perfectly, translates them into C++ BSON commands, and sends them across the internet. Furthermore, MongoDB is "schema-less" (it lets you save anything). Mongoose forces strict rules (Schemas) so bad data cannot enter your database.

Think of Mongoose as the strict bouncer for your MongoDB database. By default, MongoDB is perfectly happy to let you throw any random, unstructured data into a collection. Mongoose steps in to bring order to that chaos, and the mongoose.Schema is its rule book.

Mongoose fixes this by acting as a strict border checkpoint inside the Node.js V8 engine. Before any data is allowed to enter the TCP socket to MongoDB, Mongoose inspects it against a Schema (a strict blueprint). If the data does not perfectly match the blueprint, Mongoose rejects it and throws an error before network bandwidth is wasted.

A Mongoose Schema defines the structure, default values, and validation rules for your documents. However, a Schema on its own doesn't actually do anything to the database. It is just a blueprint.

To use a Schema, you have to compile it into a Model.
➖➖➖ The Mechanics of the Schema

=> 1. new mongoose.Schema({}): Define the Schema
Write the blueprint (what fields exist and their types). This is a constructor function. It does not talk to the database. It purely creates a Javascript object in RAM that defines the rules.

=> 2. mongoose.model('User', schema): Compile the Model
The Schema is just a blueprint; it cannot do anything. The model function is a Factory. It takes your blueprint and compiles it into a powerful Class. This compiled Class is what actually contains the functions to talk to the database (like .find(), .create(), .updateOne()). It provides the interface to query, create, update, and delete records in the database.

=> 3. Create Documents: Use the Model to create individual instances (documents) that perfectly match your Schema.


========== SchemaType Options and Validators ==========

=== 1. Schema Types (The Building Blocks)

When you define a path (a field) in a schema, you assign it a SchemaType. This tells Mongoose what kind of data to expect and how to cast it. Tells the V8 engine to strictly enforce Javascript memory types.

➖ type : String
=> String: Plain text (e.g., "Hello").
=> Number: Numeric values, including decimals (e.g., 42, 3.14).
=> Boolean: true or false.
=> Date: Standard JavaScript Date objects. Usually defaults to Date.now.
=> Buffer: Used to store binary data (like images or small files directly in the document).
=> Mixed (mongoose.Schema.Types.Mixed): A flexible data type where absolutely anything goes. Mongoose loses the ability to auto-validate or track changes deeply inside Mixed types, so use it sparingly.
=> ObjectId (mongoose.Schema.Types.ObjectId): Usually used to store the _id of another document. This is how you create relationships (joins) between different collections.
=> Array: Can be an array of standard types (e.g., [String]) or an array of nested sub-documents.
=> Decimal128: For high-precision decimals (often used for exact currency calculations where floating-point math causes rounding errors).
=> Map: A key-value object where the keys are strings, but the values must all be of the same specified Schema Type.
=> UUID: Universally Unique Identifiers (typically stored as binary data under the hood).


=== 2. Validators (The Enforcers)

For ALL Schema Types:
=> required: Boolean or function. If true, the field must be provided. If this field is missing from the user's request, Mongoose triggers a validation error.
=> default: Sets a default value if none is provided.
=> select: Boolean. If false, this field is hidden from query results by default (useful for passwords).
By setting select: false, Mongoose physically strips the password out of every single read query automatically. The only way to get it is to explicitly demand it

For String:
=> enum: An array of strictly allowed string values.
Enum stands for "Enumerated Type." It is a concept from low-level languages like C and Java that Mongoose simulates in JavaScript.
=> match: A Regular Expression the string must pass.
=> minLength / maxLength: Restricts the character count of the string.

For Number:
=> min / max: Restricts the lowest and highest allowed numerical values.
=> enum: An array of allowed numbers.


=== 3. The Modifiers (Setters)

String Modifiers:
=> trim: Removes whitespace from the beginning and end of a string.
=> lowercase: Converts the string to all lowercase letters.
=> uppercase: Converts the string to all uppercase letters.
=> custom Setter:
price: {
    type: Number,
    ➖ Custom Setter: Always rounds the number to 2 decimal places before saving
    set: v => Math.round(v * 100) / 100 
  }


=== 4. The Index Options (The unique trap)

➖ unique : true
unique is the biggest trick in Mongoose. It is not a validator. It is a helper to build a MongoDB Unique Index.

=> A validator runs in your Node.js application.
=> unique tells the actual MongoDB database to put a physical lock on that collection so no two documents can have the same value.
=> If a validation fails, Mongoose throws a ValidationError. If unique fails, MongoDB throws a native Duplicate Key Error (Error Code E11000).

Other Index-Related Options:

=> index: Boolean. Explicitly tells Mongoose to create a standard index on this field to make searching by this field much faster.
=> sparse: Boolean. Tells MongoDB to only index documents that actually contain this field (useful if you have a unique field, but some users don't have that field at all).


=== 5. Behavioral Options

These tell Mongoose how to treat the field during the application's lifecycle.

=> immutable: Boolean. If true, this field can be set when the document is created, but it can never be changed afterward. Great for things like createdAt or stripeCustomerId.
=> alias: String. Creates a virtual property that gets and sets the value of another field. Useful if you want to use a short, ugly name in the database to save space, but a clean name in your code.
=> select: Boolean. As mentioned previously, if false, Mongoose will intentionally leave this field out of query results unless you explicitly ask for it.

pwd: {
    type: String,
    unique: true, ➖ Creates a unique index in the database
    index: true   ➖ Makes querying by username lightning fast
    select: false,  ➖ Hidden by default
    alias: 'password' ➖ You can type user.password in your code, but it saves as 'pwd' in the DB
  },

➖➖➖ type:String - ValidationError
If you set type: String, the V8 engine allocates memory and allows the user to type literally anything: "Admin", "apple", "12345".
By adding enum: ['user', 'admin'], you turn that open text field into a strict Whitelist Dropdown Menu.
When data hits this field, Mongoose physically loops through array. If the user's string does not exactly match one of those specific strings, Mongoose instantly throws a ValidationError and blocks the database write. It guarantees absolute data consistency for categories, statuses, and roles.

If you send the number 12345 into current email field, Mongoose will NOT throw an error. It will successfully save it to the database.
When you wrote type: String in your Schema, you did not create a "Validator" (a rule). You created a "Caster" (a memory instruction).

When Mongoose receives the number 12345, it says:
"The Blueprint requires a String. The user gave me a Number. Before I panic and throw an error, can I mathematically convert this Number into a String?"
It runs standard V8 JavaScript coercion: String(12345). Because this perfectly converts into the text "12345", Mongoose smiles, passes the data through, and saves "12345" to MongoDB.

What if the hacker is smart, and instead of sending a Number, they send a raw JSON Object: email: { "hacked": true }?
Mongoose tries to run String({ "hacked": true }). The V8 engine completely fails to convert a complex object into a simple string.
When Mongoose physically cannot cast the data, it throws a completely different type of error. It is not a ValidationError. It is a CastError.

If you do not teach your globalErrorHandler to catch this specific C++ explosion, your user will get a massive, ugly stack trace.


===== What is "__v": 0 inside the document

This is called the Version Key (versionKey). Mongoose automatically injects this into every single document it creates.

The Physics of Concurrency:
Imagine you and another Administrator both open the exact same User document at the exact same millisecond.

You try to add a new string to an array inside that user.
The other Administrator tries to delete that entire array.
If both of your saves hit the database at the same time, the V8 engine and MongoDB could corrupt the data.

To prevent this, Mongoose uses Optimistic Concurrency Control.
When the document is created, it starts at Version 0. Every time a complex mutation happens (like modifying arrays), Mongoose silently changes it to __v: 1, then __v: 2. If your server tries to save a document, but the version number in RAM does not match the version number on the hard drive, Mongoose forcefully blocks the save and throws a VersionError.
It is an invisible shield protecting your data from simultaneous overwrite collisions.

➖➖➖➖➖  ➖➖➖➖➖

Once you have defined the shape of your data using Types and Validators, you need to define its behavior.

This is where Mongoose truly outshines native MongoDB. It allows you to attach logic directly to your schema so your data models are smart, self-contained, and reusable.

=== Category 1: The Interceptors (Middleware / Hooks)

Middleware (also called pre and post hooks) allows you to intercept processes in Mongoose. You can run code right before or right after a document is saved, updated, or deleted.

Important Rule for Middleware: Always use standard function() syntax, never arrow functions () => {}. Mongoose needs to bind this to the document or query, and arrow functions break that binding.

➖➖➖ 1. schema.pre(action, callback)

=> What it does: Runs your logic before a specific action happens.
=> When it runs: Before save, validate, remove, updateOne, find, etc.
=> What it takes: The name of the action (String), and a callback function that receives a next parameter.
=> What it modifies: Can modify the document (this) before it gets saved to the database.
=> What it returns: Nothing. You must call next() to tell Mongoose to proceed to the next middleware or the actual save action.

➖➖➖ 2. schema.post(action, callback)

=> What it does: Runs your logic after a specific action has completed.
=> When it runs: After the document is saved, found, or removed.
=> What it takes: The action name (String), and a callback function that receives the finished document (doc) and next.
=> What it modifies: Usually doesn't modify the document since it's already saved. Typically used for logging, triggering emails, or cleaning up related data.
=> What it returns: Nothing. Call next().

=== For the last 10 years, Mongoose used the next() function in its middlewares. It was a Callback Pattern. You had to manually press the next()
However, mixing Callbacks (next) with Promises (async/await) causes massive memory tracking issues in the Node.js V8 engine. If an error happened, the stack trace would break, and developers couldn't see exactly which file caused the crash.

The Mongoose 9 Revolution:
To fix this, the creator of Mongoose completely rewrote the internal C++ middleware engine for Version 9.
They explicitly deleted the next parameter from all async functions.

=== Category 2: The Computed Properties (Virtuals)

Virtuals are document properties that you can get and set, but they do not get saved to the MongoDB database.

➖➖➖ 1. schema.virtual(propertyName) - check below "The ghost field" line 345

=> What it does: Creates a "fake" field computed from actual fields.
=> When it runs: Whenever you try to read (.get()) or write to (.set()) the virtual property name.
=> What it takes: A string representing the name of the virtual field.
=> What it returns: The computed value.

const userSchema = new mongoose.Schema({
  firstName: String,
  lastName: String
});

➖ A GETTER Virtual
userSchema.virtual('fullName').get(function() {
  return `${this.firstName} ${this.lastName}`;
});

➖ A SETTER Virtual
userSchema.virtual('fullName').set(function(name) {
  const parts = name.split(' ');
  this.firstName = parts[0];
  this.lastName = parts[1];
});

➖ Usage:
const user = new User({ firstName: 'John', lastName: 'Doe' });
console.log(user.fullName); ➖ "John Doe" (runs the getter)

user.fullName = "Jane Smith"; ➖ runs the setter
➖ Now user.firstName is "Jane" and user.lastName is "Smith"

Note: By default, Mongoose strips virtuals when you convert a document to JSON (like when sending an API response). To include them, you must add { toJSON: { virtuals: true } } to your Schema options.


=== Category 3: The Custom Behaviors

These methods allow you to add custom JavaScript functions to your documents, your models, or your database queries.

➖➖➖ 1. Instance Methods: schema.methods

=> What it does: Attaches a custom function to individual documents (instances).
=> When it runs: When you explicitly call it on a fetched document.
=> The this context: Refers to the specific document.

userSchema.methods.comparePassword = async function(candidatePassword) {}

➖ Usage (must fetch a user first)
const user = await User.findOne({ email: 'test@test.com' });
const isMatch = await user.comparePassword('mySecret123');

➖➖➖ 2. Static Methods: schema.statics

=> What it does: Attaches a custom function to the Model itself.
=> When it runs: When you explicitly call it on the uppercase Model object.
=> The this context: Refers to the Model (so you can run database queries inside it).

➖ Definition
userSchema.statics.findByEmail = function(email) {
  ➖ 'this' refers to the User Model
  return this.findOne({ email: email }); 
};

➖ Usage (called on the Model, no need to fetch a specific user first)
const user = await User.findByEmail('test@test.com');


➖➖➖ 3. Query Helpers: schema.query

=> What it does: Allows you to create custom, chainable methods for your database queries.
=> When it runs: When chained onto a query like .find() or .findOne().
=> The this context: Refers to the current query object.

➖ Definition
userSchema.query.byName = function(name) {
  ➖ 'this' is the query, so we chain a regex search to it
  return this.where({ username: new RegExp(name, 'i') });
};
    ➖ where() is one of the most powerful and heavily used methods in Mongoose's Query API
    ➖ where() is an alternative approach. It allows you to build queries using a chainable, readable syntax that reads almost like an English sentence.

➖➖➖ Category: Query Method (belongs to mongoose.Query).

=> What it does: Specifies which field (or fields) you want to apply search conditions to.
=> What it returns: The Query object itself. Because it returns the query, you can immediately chain more commands onto it.

You pass everything into a single object. If you have complex conditions (greater than, less than, etc.), you have to use MongoDB operator syntax (like $gte, $lt, $in).

  ➖ Find admin users between age 18 and 65
  User.find({
    role: 'admin',
    age: { $gte: 18, $lte: 65 }
  });
  Note: This works perfectly fine, but as queries get larger, nested objects can become hard to read.

  Instead of a giant object, you declare the field using .where('fieldName'), and then chain helper methods onto it.

  ➖ Find admin users between age 18 and 65
  User.find()
    .where('role').equals('admin')
    .where('age').gte(18).lte(65);

➖ Usage (Chainable!)
const users = await User.find().byName('alice').limit(10);


Once you declare .where('someField'), you unlock a bunch of chainable methods to test that specific field. Here are the most common ones:
  .equals(value): The field must exactly match this value.
  .gt(value) / .gte(value): Greater than / Greater than or equal  to.
  .lt(value) / .lte(value): Less than / Less than or equal to.
  .in([array]): The field value must be one of the items in this array.
  .nin([array]): The field value must not be in this array (Not In).
  .regex(regularExpression): The field must match this Regex pattern.
  .exists(boolean): Checks if the field exists in the document at all.


=== Category 4: Structural Methods (Schema Configuration)

These are utility methods used to modify the schema structure itself after it has been created, or to bundle logic together.

➖➖➖ 1. schema.add(obj)
=> What it does: Adds new keys to the schema after it was initially defined.
=> Useful for: Dynamically building schemas based on external configurations.

➖➖➖ 2. schema.index(fields, options)
=> What it does: Explicitly defines database indexes at the schema level rather than the field level.
=> Useful for: Creating compound indexes (indexes that span multiple fields, like ensuring a userId and productId combination is unique).

➖ Ensures a user can only have one review per specific product
reviewSchema.index({ userId: 1, productId: 1 }, { unique: true });

➖➖➖ 3. schema.plugin(pluginFunction, options)
=> What it does: Allows you to package up any combination of the above (pre hooks, methods, virtuals) into a separate file and apply it to multiple schemas.
=> Useful for: Reusability. For example, creating a timestampPlugin that adds createdAt and updatedAt to any schema it gets attached to.


===== The Mechanics of State Tracking: How this.isModified() Works

`this` points to an intermediate, transient state: The Mongoose Document Instance residing in Node.js (V8) RAM.
To understand this, you must look at the three distinct phases of data in this pipeline:

=> The Payload (Plain Old JavaScript Object - POJO): This is req.body. It is just a raw, unvalidated block of memory parsed from the incoming HTTP request.

=> The Mongoose Document (this): When you call User.create(req.body), Mongoose takes the POJO, runs it through the Schema validators, casts the types, injects defaults (like role: 'user'), and instantiates a heavy C++ backed JavaScript Class called a Document. This is what `this` refers to. It lives entirely in your server's RAM.
Inside pre('save'), `this` is your final opportunity to intercept and mutate the fully validated V8 JavaScript object just milliseconds before it is converted to binary and sent over the network.

=> The BSON Document: After the pre('save') middleware finishes executing, Mongoose takes this, serializes it into binary format (BSON), and pushes it across the TCP network to the MongoDB hard drive.

A Mongoose Document (this) is not a static object; it is a State Machine. Mongoose utilizes internal change-tracking (often called "dirty tracking") to optimize database operations and protect data integrity.
algorithmic process of how isModified() evaluates state:
The Initial Snapshot:
When a Mongoose Document is instantiated (either brand new via create(), or pulled from the database via findOne()), Mongoose creates a hidden internal dictionary tracking the original state of every field.

The Mutation Flagging:
If you change a property in RAM (e.g., user.password = "newPassword123"), Mongoose detects the setter method being triggered. It immediately flags the password path as "modified" (or "dirty") in its internal tracking state.

The isModified(path) Evaluation:
When you call this.isModified('password'), Mongoose simply checks that internal flag.

Scenario A (New User Registration): You are creating a brand new user. All defined paths are inherently new, so Mongoose flags all of them as modified. this.isModified('password') evaluates to true. The cryptographic hashing algorithm executes.

Scenario B (Updating an Email): Imagine a user updates their email. You pull their document from the database, change this.email = "new@test.com", and call this.save(). The pre('save') hook fires again. However, because you did not touch the password, Mongoose did not flag it. this.isModified('password') evaluates to false.


===== How The Ghost field works
MongoDB is a NoSQL database. It physically does not have "Tables" or "Foreign Keys" like SQL databases do. It cannot do a true physical JOIN at the hard-drive level.

Instead, Mongoose performs an Application-Level Join. It uses a mathematical bridge called an ObjectId Reference, and a tool called .populate() to fake a SQL JOIN directly in your V8 RAM.

const reviewSchema = new mongoose.Schema({
    author: {
      type: mongoose.Schema.ObjectId,
      ref: "User", ➖ This tells Mongoose exactly WHICH model this ID belongs to!
      required: [true, "A review must belong to an author"],
    },
  },
  {timestamps: true},
);

Right now, your Review knows who its User is. The arrow points from Review -> User. This is called `Parent Referencing`.

But what if the React frontend wants to display a User's Profile Page, and they need a list of every review that specific user has ever written? The arrow doesn't point backward. The User has no idea their reviews exist.

MongoDB has a strict, hard-coded C++ limit: No single document can ever exceed 16 Megabytes.

The Golden Rule of NoSQL: Never use arrays that can grow infinitely.

The Enterprise Solution: The Virtual Populate
We need the User to have a reviews array, but we cannot physically store it on the hard drive.

To solve this, Mongoose invented a feature called Virtuals. A Virtual is a ghost field. It exists only in your Node.js V8 RAM, and completely vanishes when the server turns off.

What Exactly is a Virtual? (Virtuals vs. Middleware)
Middleware (.pre, .post): These are Timelines. They are actions that happen at specific moments in time. "Right before you save to the hard drive, hash the password."

Virtuals (.virtual): These are Computed States. They are not actions, they are math equations that calculate data on-the-fly only when asked.

=== Type 1: The Standard Virtual (Derived Math)

Imagine you have a User schema with firstName: "Anand" and lastName: "Prakash".
You want the frontend to have a fullName field.

The Bad Way: You create a fullName column in the database. Now you are wasting hard drive space storing duplicate data.

The Virtual Way:
userSchema.virtual('fullName').get(function() {
  return `${this.firstName} ${this.lastName}`;
});

=== Type 2: The Virtual Populate (Relational Math)

This is what we built for the Reviews. Instead of stitching two strings together, we are stitching two Collections together.

Let us teach the User how to summon their reviews into RAM on-demand, without ever saving an array to the database.


==> Step 1: Building the Ghost Field (User.js)

const userSchema = new mongoose.Schema({
  // ... all your user fields ...
}, {
  timestamps: true,
  toJSON: { virtuals: true }, // 🚨 TELL V8 TO CONVERT GHOSTS TO JSON
  toObject: { virtuals: true }
});

userSchema.virtual("reviews", {
  ref: "Review", // 1. What Model are we looking for?
  foreignField: "author", // 2. What is the name of the bridge on the OTHER side?
  localField: "_id", // 3. What is the name of the bridge on THIS side?
});

Here is exactly how the Mongoose Engine reads that dictionary:

ref: 'Review': "Hey Mongoose, I need you to leave the User collection and open a temporary connection to the Review collection."

localField: '_id': "Grab the _id of the User I am currently looking at (e.g., Anand's ID)."

foreignField: 'author': "Now, scan through the entire Review collection. Find every single review where the author column exactly matches Anand's ID. Grab them all, put them in an array, and attach them to the JSON response."

==> Step 2: Executing the Summoning (The Controller)
const user = await User.findById(req.user._id).populate({
    path: 'reviews', // The name of the ghost field!
    select: 'text rating -author'
  });

1. The Mystery of the Duplicate id

Mongoose has a built-in, hidden "Ghost Field" on every single Schema it creates. It creates a virtual field called `id` that simply takes the _id (which is a heavy C++ ObjectId buffer) and converts it into a lightweight Javascript String.

Normally, this id ghost is invisible when you send JSON to the frontend. But remember what we added to your User Schema in the last step?
toJSON: { virtuals: true }
You told Mongoose: "When converting this document to JSON, make ALL ghosts visible." You intended to reveal your reviews ghost, but you accidentally revealed Mongoose's default id ghost as well!

If you want to hide it, you simply tell Mongoose to turn off its default ghost. Update your Schema options
{
  timestamps: true,
  toJSON: { virtuals: true },
  toObject: { virtuals: true },
  id: false // 🚨 Kills the default string ghost
}
